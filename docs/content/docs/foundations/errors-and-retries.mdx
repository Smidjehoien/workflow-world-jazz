# Errors & Retries

In addition to the default `Error` you can throw in step functions, Workflow SDK provides two new types of errors you can use to further customize the error exit strategy when a step errors. These include [`FatalError`](/docs/api-reference/vercel-workflow-core/fatal-error) and [`RetryableError`](/docs/api-reference/vercel-workflow-core/retryable-error).

## Retrying Step Functions

When an `Error` or a `RetryableError` is thrown inside a step function, the step is automatically re-enqueued to be executed at a later time. This allows step functions to be *durable*.

It may not seem entirely obvious what the difference between `Error` and `RetryableError` is since they both retry a step function. The difference is that throwing an `Error` automatically retries a step after 1 second, while `RetryableError` has an additional parameter `retryAfter` to specify when the step function should be retried.

```ts lineNumbers
import { RetryableError } from '@vercel/workflow'; // [!code highlight]

async function callAPI(endpoint: string) {
  "use step";
  
  const response = await fetch(endpoint);

  if (response.status === 429) {
    // Retries step, optional retryAfter
    throw new RetryableError('Rate limited', { // [!code highlight]
        retryAfter: "10m" // [!code highlight]
    }); // [!code highlight]
  }
  if (!response.ok) {
    throw new Error(`HTTP ${response.status}`); // Retries step, no optional retryAfter [!code highlight]
  }
  
  return response.json();
}
```

To learn more about `RetryableError`, refer to [API Reference: `RetryableError`](/docs/api-reference/vercel-workflow-core/retryable-error).

## Failing Step Functions

Step functions can also be marked as failed and won't be re-enqueue to be retried when a `FatalError` is thrown.

```ts lineNumbers
import { FatalError } from '@vercel/workflow'; // [!code highlight] 

async function callAPI(endpoint: string) {
  "use step";
  
  const response = await fetch(endpoint);

  if (response.status === 404) {
    throw new FatalError("Resource not found") // Step will not retry [!code highlight]
  }

  return response.json();
}
```

To learn more about `FatalError`, refer to [API Reference: `FatalError`](/docs/api-reference/vercel-workflow-core/fatal-error).

Here's some examples of when you would use each:

- `Error`: Temporary step failures (network issues, rate limits, etc.).
- `FatalError`: Permanent failures (invalid data, missing resources, etc.).
- `RetryableError`: When you know the retry timing/backoff (API rate limit headers, etc.).

## Context-Aware Retries

Workflow SDK provides a function [`getStepMetadata`](/docs/api-reference/vercel-workflow-core/get-step-metadata), which can be used to retrieve the current step function metadata.

This function provides a couple of useful properties to give you more control over step retries:

- `stepId`: Unique identifier for current executing step
- `stepStartedAt`: Timestamp when step started
- `attempt`: Number of times current step has been executed

```ts lineNumbers
import { FatalError, RetryableError, getStepMetadata } from '@vercel/workflow';

async function callAPI(endpoint: string) {
  "use step"

  const { attempt, stepStartedAt, stepId } = getStepMetadata(); // Access step metadata [!code highlight]
  const response = await fetch(endpoint);

  if (response.status === 429) {
    // todo w attempt
  }

  if (response.status === 404 && attempt >= 3) {
    throw new FatalError(`[${stepId}] Resource still missing as of ${stepStartedAt.toISOString()}`);
  }

  if (!response.ok) {
    throw new Error(`HTTP ${response.status}`);
  }

  return response.json();
}

myStep.maxRetries = 10
```
