# Concepts

## What is a "durable function"?

Workflows (i.e. _durable functions_) are a programming model for building long-running, stateful application logic on top of stateless compute.

Instead of holding memory or threads open, the runtime persists progress as an event log and
deterministically replays your code to reconstruct in‑memory state after
cold starts, failures, or scale events.

This lets you write ordinary asynchronous code (awaits, loops, error handling)
while the platform provides reliable timers, retries, idempotency, and
external resumption (via webhooks).

The result is resilient function execution that can span minutes to months
(or even forever) with strong operational guarantees.

## Core Components

Workflows are regular JavaScript/TypeScript functions with two main components:

* **Workflow function** - Orchestrates execution flow
* **Step function** - Performs actual work with side effects

### Workflow Function

> Directive: `"use workflow"`

The workflow function is the orchestrator that coordinates your application logic. It defines the sequence and logic of step executions.

```ts {2}
export async function orderProcessingWorkflow(orderId: string) {
  "use workflow";
  
  // Orchestrate multiple steps
  const order = await fetchOrder(orderId);
  const payment = await processPayment(orderId);
  const shipment = await createShipment(orderId);
  
  return { orderId, status: 'completed' };
}
```

Workflows ensure:

* **Deterministic execution**: Must produce the same result when replayed
* **Multiple executions**: Re-runs from the beginning during replay
* **Sandboxed environment**: Limited to orchestration logic
* **Event sourcing**: All step results are persisted and replayed

<Callout type="important">
Any code that needs Node.js APIs, npm packages, or produces side effects must be placed in step functions. The workflow function should only contain orchestration logic.
</Callout>

### Step Function

> Directive: `"use step"`

Step functions contain the actual implementation logic with full Node.js runtime access.

```ts {2}
async function fetchOrder(orderId: string) {
  "use step";
  
  // Full Node.js access - use any npm package
  const stripe = new Stripe(process.env.STRIPE_KEY);
  
  const payment = await stripe.charges.create({
    amount: order.total,
    currency: 'usd',
    source: order.paymentToken
  });
  
  return { paymentId: payment.id, status: 'paid' };
}
```

Steps have:

* **Full Node.js access**: Use any API or npm package
* **Automatic retries**: Retry upon failure
* **Error handling**: Control retry behavior with error types

## Writing Workflows

### Basic Structure

A simple workflow shows the difference between orchestration and execution:

```ts
// Step function - performs actual work
async function greet(name: string) {
  "use step";
  
  // Can use Node.js APIs and npm packages
  const message = `Hello ${name} at ${new Date().toISOString()}`;
  console.log(message);
  return message;
}

// Workflow function - orchestrates steps
export async function helloWorldWorkflow(userName: string) {
  "use workflow";
  
  const greeting = await greet(userName);
  return { greeting, completed: true };
}
```

### Control Flow Patterns

#### Sequential Execution

Steps execute one after another:

```ts
export async function dataProcessingWorkflow(data: any) {
  "use workflow";
  
  // Each step waits for the previous to complete
  const validated = await validateData(data);
  const processed = await processData(validated);
  const stored = await storeResults(processed);
  
  return { id: stored.id, status: 'complete' };
}
```

#### Parallel Execution

Steps execute simultaneously for better performance:

```ts
export async function parallelWorkflow(userId: string) {
  "use workflow";
  
  // All three execute at the same time
  const [user, orders, preferences] = await Promise.all([
    fetchUser(userId),
    fetchOrders(userId),
    fetchPreferences(userId)
  ]);
  
  return { user, orders, preferences };
}
```

#### Race Conditions

Steps race to either timeout or produce a result:

```ts
export async function timeoutWorkflow(taskId: string) {
  "use workflow";
  
  // First to complete wins
  const result = await Promise.race([
    performTask(taskId),
    timeout(30000) // 30 second timeout
  ]);
  
  return result;
}
```

### Error Handling

Workflow SDK provides specialized error classes for control flow:

```ts
import { FatalError, RetryableError } from '@vercel/workflow-core';

async function reliableAPICall(endpoint: string) {
  "use step";
  
  const response = await fetchStep(endpoint);
  
  if (response.status === 404) {
    // Don't retry - permanent failure
    throw new FatalError('Resource not found');
  }
  
  if (response.status === 429) {
    // Retry later - temporary failure
    throw new RetryableError('Rate limited');
  }
  
  if (!response.ok) {
    // Default behavior - will retry
    throw new Error(`HTTP ${response.status}`);
  }
  
  return response.json();
}
```

<Callout type="info">
To learn more about error classes, refer to the [API Reference](./api-reference#error-classes).
</Callout>

### Long-Running Workflows

Workflows can run for extended periods using the `sleep` function:

```ts
import { sleep } from '@vercel/workflow-core';

export async function scheduledWorkflow(userId: string) {
  "use workflow";
  
  // Send welcome email immediately
  await sendWelcomeEmail(userId);
  
  // Wait 1 day
  await sleep('1d');
  
  // Send follow-up
  await sendFollowUpEmail(userId);
  
  // Wait 1 week
  await sleep('7d');
  
  // Send survey
  await sendSurveyEmail(userId);
}
```

## Best Practices

### 1. Workflows Are Deterministic

```ts
// ❌ Bad - Non-deterministic in workflow
export async function badWorkflow() {
  "use workflow";
  const id = Math.random(); // Changes on replay, different each time!
}

// ✅ Good - Non-deterministic in step
async function generateId() {
  "use step";
  return { id: Math.random(), timestamp: Date.now() };
}

export async function goodWorkflow() {
  "use workflow";
  const { id, timestamp } = await generateId();
}
```
