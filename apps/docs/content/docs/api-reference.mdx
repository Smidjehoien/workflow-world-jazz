# API Reference

## @vercel/workflow-core

The core package contains the functions and primitives that are used within workflows.

### Core Functions

#### `getContext`

Returns the current workflow or step execution context. This is a context-safe accessor that works in both workflow and step functions.

```ts
import { getContext } from '@vercel/workflow-core';
```

<TSDoc
  definition={generateDefinition({
    code: `
import { getContext } from "@vercel/workflow-core"
export default getContext
    `
  })}
  typeLinkMap={{
    "WorkflowContext": "api-reference#workflowcontext"
  }}
/>

<Callout type="info">
The `stepId`, `stepStartedAt`, and `attempt` properties are only available inside step functions.
</Callout>

#### `sleep`

Sleep within a workflow for a given duration. This is deterministic and replay-safe.

```ts
import { sleep } from '@vercel/workflow-core';
```

<TSDoc
  definition={generateDefinition({
    code: `

    `
  })}
/>

#### `getWebhook`

Registers a webhook URL that can be provided to external systems outside of the workflow run (such as sending in an email for human-in-the-loop workflows).

```ts
import { getWebhook } from '@vercel/workflow-core';
```

<TSDoc
  definition={generateDefinition({
    code: `

    `
  })}
/>

#### `createWorkflowOutputStream`

Retrieves the output writable stream associated with the current workflow run.

Call this inside a workflow function to obtain a `WritableStream` you can pass into steps but it can't be called inside a step.

```ts
import { createWorkflowOutputStream } from '@vercel/workflow-core';
```

### Runtime Functions

These functions are used outside of workflow and step functions and are accessed from the runtime entrypoint e.g. where `start(workflowFn)` is called.

#### `getWorkflowOutputStream`

Retrieves the implicit readable stream for a workflow run. Use this in your application or API routes to consume streaming output by `runId`.

```ts
import { getWorkflowOutputStream } from '@vercel/workflow-core/runtime';
```

### Error Classes

#### `FatalError`

Throw this error in a step function to indicate a permanent failure that should not be retried.

<TSDoc
  definition={generateDefinition({
    code: `
import { FatalError } from '@vercel/workflow-core';
export default FatalError
    `
  })}
/>

#### `RetryableError`

Throw this error in a step function to indicate a temporary failure that should be retried. When a step function throws a `RetryableError`, the workflow runtime will automatically retry the step up to 3 times before marking it as failed.

<TSDoc
  definition={generateDefinition({
    code: `
import { RetryableError } from '@vercel/workflow-core';
export default RetryableError
    `
  })}
/>

<Callout>
Always use `FatalError` for permanent failures (like validation errors) and `RetryableError` for temporary issues (like network failures or rate limits). Step functions will automatically retry up to 3 times when a `RetryableError` is thrown.
</Callout>

### Next.js Integration

For Next.js applications, you can use the `@vercel/workflow-next` package which provides the `withWorkflow()` configuration wrapper. See the [Getting Started guide](/docs/getting-started) for setup instructions.
