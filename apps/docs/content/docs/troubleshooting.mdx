# Troubleshooting

Common issues and solutions when working with Workflow SDK.

## Build & Setup Issues

### Workflows not being detected

**Problem:** Your workflows aren't being compiled or recognized.

**Solution:**

- Ensure your workflow functions and step functions contain the `"use workflow"` or `"use step"` directives to be processed
- Check that `withWorkflow()` wraps your Next.js config

### Build errors with external packages

**Problem:** External packages cause weird build time or runtime errors in steps.

**Solution:** Add problematic packages to `serverExternalPackages` in your Next.js config.

For example with `@node-rs/xxhash`:

```ts
const nextConfig: NextConfig = {
  serverExternalPackages: ['@node-rs/xxhash'],
};
```

## Runtime Issues

### "Cannot be called outside workflow/step" errors

**Problem:** Getting errors about functions only available in workflow context.

**Solution:**

- `getContext()` can only be called inside a workflow or step function
- `sleep()` and `getWebhook()` only work inside workflow functions
- Move the code inside the appropriate function with the correct directive

## Development Best Practices

### File organization

**Recommended structure:**

<FileTree>
  <FileTree.Folder name="workflows" open>
    <FileTree.File name="user-signin-workflow.ts" />
    <FileTree.File name="user-signup-workflow.ts" />
    <FileTree.Folder name="steps" open>
      <FileTree.File name="api.ts" />
      <FileTree.File name="database.ts" />
      <FileTree.File name="email.ts" />
    </FileTree.Folder>
  </FileTree.Folder>
</FileTree>

**Why:** Separating workflows and steps prevents invalid code from sneaking into generated bundles.

### Local development setup

For local development with Next.js:

1. Install dependencies with the correct package manager
2. Wrap your Next.js config with `withWorkflow()`
3. Run `pnpm dev` (or your package manager's dev command)
4. The embedded transport handles local workflow execution

### Debugging workflows

**Tips for debugging:**

- Use `console.log()` in step functions for debugging
- Check `next dev` server logs for local development or Vercel logs in production
- Test steps in isolation before combining in workflows

## Common Patterns

### Handling non-deterministic code

**Problem:** Using `Math.random()` or `Date.now()` in workflow functions can have un-predictable behavior.

**Solution:** Move non-deterministic code to step functions:

```ts
// Bad - in workflow
export async function myWorkflow() {
  "use workflow";
  const id = Math.random(); // Seeded per workflow
  const now = Date.now();   // Fixed, then incremented throughout lifetime of workflow run after completed steps
}

// Good - in step
async function generateId() {
  "use step";
  return Math.random(); // Unseeded, normal behavior
}

export async function myWorkflow() {
  "use workflow";
  const id = await generateId();
}
```

### Testing workflows

**Approach:** Test steps individually, then test workflow orchestration:

```ts
// Test individual steps
import { myStep } from './workflows/steps/myStep';

test('myStep processes data correctly', async () => {
  const result = await myStep(testData);
  expect(result).toBe(expected);
});
```
